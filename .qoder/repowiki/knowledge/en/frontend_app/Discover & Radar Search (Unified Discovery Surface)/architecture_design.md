Three-file leaf module split by concern:
- `src/pages/Discover.tsx` is the React entry: owns tab state (`quick | scene | vibe`), URL sync via `useSearchParams`, infinite-scroll result grid driven by `@tanstack/react-query`'s `useInfiniteQuery` calling `api.searchMulti`, and delegates the two advanced modes to sibling components `SceneSearch` and `VibeSearch`.
- `src/services/discovery.ts` is a pure client-side intent parser and scorer. It normalizes Portuguese input, classifies it into `quick | clue | vibe` via keyword markers, extracts media-type filters, positive/negative tags, preferred length/pace/year, then scores candidate `MediaItem[]` against those signals plus user profile/library data; exports `classifySearch`, `parseDiscoveryIntent`, `rankCandidates`, `rankQuickPick`, `explainHype`.
- `src/services/radarSearch.ts` implements the "Radar" identification pipeline: OCR via lazy-loaded `tesseract.js` (`recognizeText`), catalog deduped search from extracted text (`searchCatalogFromText`), optional remote anime frame matching against `trace.moe` (`searchAnimeFrame`), scene-description fallback (`searchSceneDescription`), and shared-link resolver that parses TMDB/AniList/MAL/Open Library/Google Books/Steam URLs into catalog IDs (`resolveSharedLink`). All branches return a uniform `RadarCandidate[]` with confidence/reason/source so the UI can merge results.

Dependency direction: `Discover.tsx` → `discovery.ts` (for routing classification) and `SceneSearch`/`VibeSearch` (which in turn call `radarSearch.ts`); both services depend only on `@/services/api`, `@/types`, and `@/services/identity.normalizeText`. No cross-import between the two service files — they are independent pipelines sharing the `MediaItem` contract.
